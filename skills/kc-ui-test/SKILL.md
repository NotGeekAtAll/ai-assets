# kc-ui-test

你的任务是基于 Playwright MCP 协助完成 UI 自动化测试工作。

## 目标

当需求涉及 UI 自动化测试时，先根据页面功能、交互流程和风险点，输出一份更完善的自动化测试文档；
只有在用户明确确认后，才进入后续的自动化测试执行、脚本生成或回归验证阶段。

## 适用场景

以下情况适合使用本 Skill：

- 用户明确要求补充 UI 自动化测试
- 页面改动涉及关键主流程
- 存在高频回归场景
- 交互链路较长，适合用自动化覆盖
- 需要基于 Playwright MCP 进行页面操作、验证与回归

## 工作原则

### 1. 先文档，后执行
默认先输出自动化测试文档，不要直接开始执行自动化测试。

### 2. 先覆盖主流程，再考虑扩展场景
优先覆盖：
- 核心业务主链路
- 高风险交互
- 容易回归出问题的流程

### 3. 测试要可执行、可验证、可维护
测试文档不能只写“测试按钮是否可点击”，而应明确：
- 前置条件
- 操作步骤
- 断言点
- 数据要求
- 风险说明

### 4. 不盲目扩大覆盖范围
优先测试最值得自动化的部分，不要机械追求全量自动化。

## 执行流程

### 第一步：理解测试背景
先识别并整理：

- 测试对象是什么页面/模块
- 改动点是什么
- 哪些流程最关键
- 是否有账号、环境、测试数据前提
- 是否已有可复用脚本或测试约定

### 第二步：判断是否适合自动化
对每个候选场景进行判断：

适合自动化的场景通常包括：
- 稳定、重复执行频率高
- 核心主流程
- 断言清晰
- 人工回归成本高

不适合优先自动化的场景通常包括：
- 频繁变化中的临时页面
- 强依赖人工感知的视觉细节
- 前置依赖复杂且难以稳定构造
- 一次性验证场景

### 第三步：输出 UI 自动化测试文档
在未得到用户确认前，先输出完整测试文档。

输出内容应包括：

#### 一、测试目标
- 本次为什么需要 UI 自动化
- 希望覆盖哪些风险

#### 二、测试范围
- 涉及哪些页面、模块、功能点
- 明确包含与不包含的范围

#### 三、测试前置条件
- 测试环境
- 账号角色
- 测试数据
- 依赖接口/开关/配置

#### 四、测试场景清单
每条场景建议包含：
- 场景名称
- 优先级（高/中/低）
- 前置条件
- 操作步骤
- 预期结果
- 关键断言点
- 是否建议自动化

#### 五、建议自动化覆盖范围
说明：
- 哪些场景建议优先自动化
- 哪些场景先手工验证
- 原因是什么

#### 六、执行建议
例如：
- 是否建议拆分为 smoke / regression
- 是否建议按模块拆文件
- 是否建议复用已有登录、导航、数据准备步骤

#### 七、风险与注意事项
例如：
- 选择器稳定性
- 测试数据污染
- 异步等待
- 环境差异
- 弹窗、权限、网络波动等问题

### 第四步：等待用户确认
文档输出后，必须等待用户确认。
在用户未确认前，不要默认开始实际自动化执行。

### 第五步：确认后再执行
如果用户明确确认要执行，再进入后续阶段，例如：

- 基于 Playwright MCP 执行页面操作与验证
- 生成 Playwright 自动化测试脚本草案
- 按测试文档逐条执行验证
- 输出执行结果、失败项与问题定位建议

## 输出格式

在“规划阶段”，请严格按以下结构输出：

### 一、测试目标
### 二、测试范围
### 三、测试前置条件
### 四、测试场景清单
### 五、建议自动化覆盖范围
### 六、执行建议
### 七、风险与注意事项
### 八、待确认项

在“执行阶段”，请按以下结构输出：

### 一、执行范围
### 二、执行结果概览
### 三、通过项
### 四、失败项
### 五、问题分析
### 六、后续建议

## 特别要求

- 如果信息不足，先基于现有信息做合理测试规划，不要直接跳过
- 不要在未确认前直接执行自动化
- 文档应以“可直接给测试/前端同事落地”为标准
- 对关键流程优先给出高质量断言点
- 若已有现成测试，可优先建议复用与补充，而不是完全重写
