# 提交工作

## 目标
让提交易于审查且安全上线：
- 只包含预期变更
- 提交按逻辑范围划分（需要时拆分）
- 提交信息描述变更内容和原因

## 需要询问的输入（若缺失）
- 单次提交还是多次提交？（如果不确定：当有不相关的变更时，默认拆分为多个小提交。）
- 提交风格：必须使用 Conventional Commits。
- 任何规则：最大主题长度、必需的作用域。

## 工作流程（检查清单）
1) 在暂存前检查工作树
   - `git status`
   - `git diff`（未暂存）
   - 如果变更较多：`git diff --stat`

2) 确定提交边界（需要时拆分）
   - 按以下维度拆分：特性 vs 重构、后端 vs 前端、格式化 vs 逻辑、测试 vs 生产代码、依赖更新 vs 行为变更。
   - 如果单个文件中有混合变更，计划使用补丁暂存。

3) 只暂存属于下一个提交的内容
   - 混合变更优先使用补丁暂存：`git add -p`
   - 要取消暂存某个块/文件：`git restore --staged -p` 或 `git restore --staged <路径>`

4) 审查实际将要提交的内容
   - `git diff --cached`
   - 健全性检查：
     - 无密钥或令牌
     - 无意外调试日志
     - 无无关的格式变动

5) 用 1-2 句话描述暂存的变更（在编写信息之前）
   - “变更了什么？” + “为什么？”
   - 如果无法清晰描述，说明提交可能太大或混杂，返回步骤 2。

6) 编写提交信息
   - 必须使用 Conventional Commits：
     - `类型(作用域): 简短摘要`
     - 空行
     - 正文（变更内容和原因，而非实现日记）
     - 脚注（如有 `BREAKING CHANGE`）
   - 多行信息优先使用编辑器：`git commit -v`
   - 如有需要，可参考 `commit-message-template.md`。

7) 运行最小相关验证
   - 在继续之前，运行仓库中最快的有效检查（单元测试、lint 或构建）。

8) 重复上述步骤直到工作树干净

## 交付物
提供：
- 最终的提交信息
- 每个提交的简短摘要（变更内容和原因）
- 用于暂存/审查的命令（至少包括 `git diff --cached`，以及运行过的任何测试）
