# kc-review

## 目标

在 Git Commit 前，对当前任务实现进行质量门禁检查。

Review 依据：

-   当前任务 `prd.md`
-   项目 Specs / Guidelines
-   当前代码 Diff

目标：

-   确认需求是否完成
-   确认实现是否符合规范
-   确认是否达到提交条件

------------------------------------------------------------------------

## 执行流程

### 1. 获取上下文

确认：

-   当前任务 `prd.md`
-   项目 Specs
-   当前代码改动

若缺少 PRD 或无法确认目标，先询问用户。

------------------------------------------------------------------------

### 2. 获取代码 Diff

根据 Git 状态选择 Review 范围。

未提交：

``` bash
git diff HEAD
git diff --staged
```

已提交：

``` bash
git fetch origin master

git diff origin/master...HEAD
```

只 Review 当前任务相关改动。

------------------------------------------------------------------------

# Review Quality Gate

## 1. PRD 符合性

检查：

-   是否完成 PRD 功能
-   是否满足验收要求
-   是否遗漏关键流程
-   是否存在无关扩展

------------------------------------------------------------------------

## 2. 功能质量

检查：

-   主流程是否正确
-   Loading / Empty / Error 是否完整
-   权限、异常、边界情况是否处理
-   是否存在明显 Bug 或回归风险

------------------------------------------------------------------------

## 3. 规范符合性

结合项目 Specs 检查：

-   是否复用已有能力
-   是否符合组件规范
-   是否符合目录和命名规范
-   是否符合 TypeScript 规范
-   是否存在重复实现

------------------------------------------------------------------------

## 4. 代码质量

检查：

-   是否容易维护
-   是否存在明显复杂实现
-   是否存在调试代码
-   是否存在性能或兼容风险

------------------------------------------------------------------------

# 问题分级

## \[必须修复\]

阻塞提交：

-   功能未完成
-   Bug
-   高风险问题
-   严重违反规范

## \[建议修复\]

建议处理：

-   可维护性问题
-   重复代码
-   潜在风险

## \[可选优化\]

后续优化：

-   可读性
-   重构建议
-   性能优化

------------------------------------------------------------------------

# 输出格式

``` markdown
## Review Quality Gate

### 必须修复

1. 问题：
   影响：
   建议：

### 建议修复

1. 问题：
   原因：
   建议：

### 可选优化

1. 建议：

## 结论

- [ ] FAIL：修改后重新 Review
- [ ] PASS：允许 Git Commit

风险：

下一步：
```

无问题：

``` markdown
## Review Quality Gate

未发现阻塞问题。

## 结论

- [x] PASS：允许 Git Commit
```

------------------------------------------------------------------------

# Trellis 联动

Review 发现重复出现、可自动检查的问题：

应沉淀到对应 Spec，避免后续重复 Review。

原则：

-   PRD 判断「是否做对事情」
-   Specs 判断「是否正确做事情」
-   Diff 判断「实现质量」
