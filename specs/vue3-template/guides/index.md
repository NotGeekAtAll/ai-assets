# 思考指南（Thinking Guides）

> **目的**：拓展你的思考范围，帮助你发现那些可能被忽略的问题。

---

## 为什么需要思考指南？

**大多数 Bug 和技术债，并不是因为能力不足，而是因为“没有想到”。**

例如：

- 没有考虑层与层之间的边界 → 导致跨层 Bug
- 没有意识到代码模式重复 → 到处都是重复代码
- 没有考虑边界情况 → 运行时错误
- 没有考虑未来维护者 → 代码难以理解和维护

这些指南能够帮助你**在编码之前，提出正确的问题**。

## 快速参考：何时需要思考？

### 什么时候需要考虑跨层问题

当满足以下任意情况时：

- [ ] 功能涉及 3 层以上（如 API、Service、Component、Database）
- [ ] 数据格式会在不同层之间发生变化
- [ ] 同一份数据需要被多个消费者使用
- [ ] 不确定某段逻辑应该放在哪一层
- [ ] 新增事件类型（Event Kind）、JSONL 记录、RPC Payload 或配置字段
- [ ] UI 或命令层开始直接对原始 Payload 字段进行类型转换（Cast）

→ 请阅读 [跨层设计思考指南](./cross-layer-thinking-guide.md)

---

### 什么时候需要考虑代码复用

当满足以下任意情况时：

- [ ] 正在编写与已有代码非常相似的逻辑
- [ ] 发现同一种模式已经重复出现 3 次以上
- [ ] 需要在多个地方同时新增同一个字段
- [ ] **正在修改任何常量（Constant）或配置（Config）**
- [ ] **准备创建新的工具函数（Utility / Helper）** ← 请先搜索是否已有类似实现！
- [ ] 两个文件都在读取同一个未类型化（Untyped）的 Payload 字段，并分别进行本地类型转换（Cast）
- [ ] 多个分支都根据 `kind` / `action` 更新同一份派生状态（Derived State）

→ 请阅读 [代码复用思考指南](./code-reuse-thinking-guide.md)

---

### 在验证 AI Code Review 结果时

当满足以下情况时，请进行额外验证：

- [ ] Reviewer 声称“用户输入可能存在恶意内容” → 请先确认数据来源（内部 Manifest？用户配置？外部 API？）
- [ ] Reviewer 提示“缺少校验” → 数据是否其实来自可信的内部来源？
- [ ] Reviewer 认为“行为发生了变化” → 请阅读代码注释，这是否属于设计上的刻意行为？
- [ ] Reviewer 指出测试存在 Bug → 在脑海中先假设该功能不存在，测试还能通过吗？如果可以，那么它很可能只是一个**自证式（Tautological）测试**。

**AI Reviewer 常见误报（False Positive）模式：**

1. **信任边界混淆（Trust Boundary Confusion）**：将内部数据（例如内置 JSON Manifest）误判为不可信的外部输入。
2. **忽略设计注释（Ignoring Design Comments）**：把代码注释中明确说明的设计行为误认为 Bug。
3. **变量理解错误（Variable Misreading）**：没有追踪变量真正的定义，例如误把以路径（Path）为 Key 的 Map 理解成以名称（Name）为 Key。

**验证原则：**

每一个 **CRITICAL** 或 **WARNING** 级别的问题，都必须结合实际代码进行验证后，再决定其优先级。

经验上，可以预留约 **35%** 的 AI Review 误报率。

## 修改前规则（重要）

> **在修改任何值之前，请务必先搜索！**

```bash
# 搜索即将修改的值
grep -r "待修改的值" .
```

这个简单的习惯，可以避免绝大多数「忘记同步更新其他地方」的问题。

## 如何使用本目录

1. **编码前**：快速浏览相关的思考指南。
2. **编码过程中**：如果感觉某段逻辑开始变得重复或复杂，请查看对应指南。
3. **遇到 Bug 后**：将新的经验总结补充到相应的思考指南中，持续沉淀团队知识。
