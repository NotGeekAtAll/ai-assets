# 代码复用思考指南（Code Reuse Thinking Guide）

> **目的**：在编写新代码之前，先停下来思考：**这段代码真的需要重新写吗？**

---

## 为什么需要思考？

重复代码是造成代码不一致和技术债的主要来源之一。

它通常会导致：

- Bug 修复无法同步到所有地方
- 相同行为随着时间逐渐产生差异
- 维护成本越来越高
- 新人越来越难理解代码结构

---

## 第一步：先搜索

在创建以下内容之前，请先搜索：

- 工具函数（Utility）
- Helper 函数
- 常量（Constant）
- 类型转换（Decoder / Type Guard）
- 组件（Component）

```bash
# 搜索是否已有类似函数
grep -r "functionName" .

# 搜索是否已有类似逻辑
grep -r "keyword" .
```

---

## 编写前先问自己

开始编码前，请先回答几个问题：

- 是否已经存在类似实现？
- 我是不是正在复制另一份代码？
- 这是第三次出现相同的模式了吗？
- 是否应该提取成公共工具？
- 项目中是否已经有统一的实现方式？

如果其中任何一个答案是 **是**，优先考虑**复用或扩展已有实现**，而不是重新创建。

---

## 常见触发场景

当出现以下情况时，请思考是否应该复用：

- 创建新的工具函数或 Helper
- 新增常量或配置项
- 在多个地方读取同一个 Payload 字段
- 创建一个与现有组件非常相似的新组件
- 根据 `kind`、`action`、`status` 等字段更新状态
- 在多个文件中进行相似的修改

---

## 代码复用原则

### 优先复用，而不是复制

不要复制已有逻辑。

优先考虑：

- 提取公共工具函数
- 扩展已有组件
- 统一使用公共常量
- 共用 Decoder、Type Guard 或 Normalizer

---

### 数据解析应只有一个入口

如果多个地方都在解析同一个未类型化（Untyped）的 Payload，例如：

```ts
const description = (ev as { description?: string }).description;
```

不要在多个地方重复进行类型转换。

应该抽取公共的：

- Decoder
- Type Guard
- Normalizer
- Projection

让数据契约只有一个定义位置。

---

### 状态流转应集中管理

如果状态由以下字段驱动：

- `action`
- `kind`
- `status`
- `phase`

应优先使用一个 Reducer（或 Dispatcher）统一管理状态流转，而不是把 `if / else` 分散在多个地方。

状态变化应该只有一个可信来源（Single Source of Truth）。

---

## 提交前检查

提交代码前确认：

- [ ] 已搜索是否存在类似实现
- [ ] 没有不必要的复制粘贴
- [ ] 常量没有重复定义
- [ ] Payload 解析已统一
- [ ] 状态流转逻辑集中管理

---

## 关联思考

如果本次修改还涉及以下情况，请继续阅读对应指南：

- 涉及多个层（API、Service、UI 等）→ **Cross-Layer Thinking Guide**
- 修改常量或配置 → **先搜索所有引用**
- 同时修改多个文件 → **确认是否可以进一步抽象**

---

## 核心原则

> **先复用，再新增；先搜索，再编码。**

花几分钟寻找已有实现，通常能节省数小时的维护和调试时间。
