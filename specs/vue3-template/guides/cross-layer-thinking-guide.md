# 跨层思考指南（Cross-Layer Thinking Guide）

> **目的**：在实现涉及多个层的功能之前，先理清数据如何在各层之间流转。

---

## 为什么需要思考？

大多数 Bug 并不是发生在某一层内部，而是发生在**层与层之间**。

例如：

* 数据格式不一致
* 字段丢失或转换错误
* 校验逻辑重复
* 多层实现了相同业务逻辑

---

## 什么时候需要阅读本指南？

当满足以下任意情况时：

* 功能涉及 3 层以上（API、Service、Database、UI 等）
* 数据需要在多个层之间转换
* 多个模块需要消费同一份数据
* 不确定某段逻辑应该放在哪一层
* 新增事件（Event）、RPC、JSON、配置等跨层数据结构

---

## 第一步：画出数据流

先明确数据如何流动：

```text
输入
  ↓
接口/API
  ↓
业务处理
  ↓
数据存储
  ↓
数据读取
  ↓
前端展示
```

对于每一步，都问自己：

* 输入是什么？
* 输出是什么？
* 谁负责转换？
* 谁负责校验？

---

## 第二步：识别边界

重点关注每一个层级边界：

| 边界                    | 常见问题         |
| --------------------- | ------------ |
| API ↔ Service         | 字段缺失、类型不一致   |
| Service ↔ Database    | 数据转换、空值处理    |
| Backend ↔ Frontend    | 序列化、日期格式     |
| Component ↔ Component | Props、数据结构变化 |

边界越清晰，Bug 越少。

---

## 跨层原则

### 每个数据只有一个 Owner

不要让多个地方分别定义：

* 数据格式
* Payload 解析
* Type Guard
* Normalizer

数据契约应该只有一个来源（Single Source of Truth）。

---

### 校验只做一次

不要每一层都做相同的校验。

应明确：

* 谁负责输入校验
* 谁负责业务校验
* 谁负责展示处理

避免重复校验或遗漏校验。

---

### 每层只负责自己的职责

不要出现：

* UI 知道数据库结构
* Component 解析原始 JSON
* 展示层承担业务逻辑

每一层只依赖相邻层，而不是跨层依赖。

---

## 提交前检查

* [ ] 数据流是否完整？
* [ ] 每个边界的数据格式是否明确？
* [ ] 是否只有一个地方负责数据转换？
* [ ] 是否只有一个地方负责数据校验？
* [ ] 是否存在重复实现相同逻辑？

---

## 关联思考

如果本次修改还涉及：

* 多处相似代码 → **Code Reuse Thinking Guide**
* 修改配置或常量 → **先搜索所有引用**
* 新增公共数据结构 → **统一维护数据契约**

---

## 核心原则

> **先设计数据流，再开始编码。**

越早理清层与层之间的职责，后续修改和维护成本就越低。
