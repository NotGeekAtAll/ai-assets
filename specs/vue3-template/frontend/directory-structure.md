# Directory Structure

> Vue3 + TypeScript 项目的目录结构规范骨架。

---

## 适用范围

本文档用于记录目标项目的前端目录结构、模块边界、文件命名和新增代码放置规则。

AI 填充时必须以真实项目结构为准，不要把下面的参考目录直接当作事实。

---

## AI 填充任务

- 读取 `src/`、`package.json`、路由配置、别名配置和构建配置。
- 找出页面、业务模块、通用组件、组合式函数、状态管理、接口请求、类型定义、静态资源的真实位置。
- 记录新增文件应该放在哪里，以及哪些目录禁止继续扩张。
- 如果同类文件分散在多个目录，说明当前现状并标记待项目确认。

---

## 参考目录骨架

```text
src/
|-- assets/          # 静态资源：图片、字体、全局样式等
|-- components/      # 跨页面复用的通用组件
|-- composables/     # 可复用的组合式逻辑
|-- constants/       # 常量和枚举映射
|-- layouts/         # 页面布局组件
|-- router/          # 路由定义和路由守卫
|-- services/        # API 请求、数据访问和外部服务封装
|-- stores/          # 全局状态管理
|-- styles/          # 全局样式、变量、主题
|-- types/           # 跨模块共享类型
|-- utils/           # 无 UI、无副作用的工具函数
|-- views/           # 路由页面
|-- App.vue
|-- main.ts
```

待项目确认：

- 实际是否使用 `views/`、`pages/` 或按业务模块组织页面。
- 实际是否使用 `stores/`、`modules/`、`features/` 等业务分层。
- 实际别名是否为 `@/`，以及是否还有其他路径别名。

---

## Module Organization

待 AI 按项目事实补充：

- 新业务模块的目录结构。
- 页面、组件、Composable、Store、Service、Types 之间的引用方向。
- 跨模块复用代码的沉淀位置。
- 业务私有组件和全局通用组件的区分标准。

---

## Naming Conventions

待 AI 按项目事实补充：

- Vue 组件文件命名：例如 `PascalCase.vue` 或 `kebab-case.vue`。
- Composable 命名：例如 `useXxx.ts`。
- Store 命名：例如 `useXxxStore.ts`。
- Service/API 文件命名。
- 类型文件和常量文件命名。

---

## Recommended Checks

- 新增文件是否放在已有同类目录中。
- 是否出现跨层直接依赖，例如组件直接解析原始接口 payload。
- 是否把业务私有代码放入全局共享目录。
- 是否存在重复的工具函数、常量、类型定义。

---

## 禁止伪造

- 不要新增项目不存在的目录作为既定规范。
- 不要声称项目使用某个状态管理、路由或请求库，除非配置或代码中能确认。
- 不要用参考目录覆盖目标项目的真实目录。
